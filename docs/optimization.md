# 智能文档问答助手优化建议

## 1. 文档目标

本文档用于整理当前项目在完成 MVP 之后，下一阶段最值得推进的优化方向，帮助后续开发时优先处理真正影响验收质量、演示效果和系统稳定性的部分。

本文档不替代 [prd.md](/d:/documentD/works/AgenticEngineering/FileManagement/docs/prd.md)、[tech.md](/d:/documentD/works/AgenticEngineering/FileManagement/docs/tech.md) 和 [test.md](/d:/documentD/works/AgenticEngineering/FileManagement/docs/test.md)，而是作为补充性的优化清单使用。

## 2. 优化原则

1. 优先提升问答效果和演示体验，而不是盲目增加新功能。
2. 优先处理容易在验收和演示中暴露的问题。
3. 优先选择投入较小、收益明确的改进项。
4. 所有优化都应服务于“更稳定、更可信、更易交付”。

## 3. 优化建议总览

| 优化方向 | 当前问题 | 建议动作 | 预期收益 |
| --- | --- | --- | --- |
| 问答准确性 | 模型可能脱离检索片段自由发挥，出现编造内容 | 给 RAG prompt 增加明确约束：只能基于检索片段回答，证据不足时要明确说明不知道或信息不足，不能自行补事实 | 降低幻觉，提升答案可信度 |
| 检索质量 | 分块策略、召回数量和检索方式可能不适配所有文档 | 调整 chunk 大小、overlap、top-k，并尝试关键词检索和向量检索的混合策略 | 提高召回质量，减少答非所问 |
| 多轮对话 | 历史消息持续累积后，prompt 可能过长、成本升高、效果下降 | 控制上下文长度，只保留最近若干轮有效对话，必要时做摘要压缩 | 降低 token 开销，保持追问质量 |
| 文档状态 | 上传到可问答之间的状态不够细，用户不容易判断进度 | 明确文档状态流转：上传中、解析中、可用、失败，并在前后端统一展示 | 提升可理解性，减少误操作 |
| Provider 可用性 | 用户难以判断某个 provider 到底支持聊天、embedding 还是只支持部分能力 | 在配置页明确提示聊天模型和 embedding 模型的支持情况，并标清回退策略 | 降低配置失败率，提升可用性 |
| 可观测性 | 出现检索回退、调用失败、来源不命中时，排查成本高 | 后端增加基础日志，记录使用了哪个 provider、检索到了哪些 chunk、为什么回退关键词检索 | 更容易定位线上和演示问题 |
| 数据安全 | `api_key` 如果直接明文存储，容易在验收时被质疑安全性不足 | 对 `api_key` 做加密存储，至少也要在文档中单独说明当前安全边界和改进方向 | 降低安全风险，提升交付说服力 |
| 部署体验 | 虽然已支持 Docker，但“一条命令启动 + 一条命令验收”的体验还可以更清晰 | 在 README 中补充最短启动命令、健康检查命令和验收步骤 | 让评审和演示者更快跑通系统 |

## 4. 分项建议

### 4.1 问答准确性

目标是让系统回答更可信，而不是只是“像回答了问题”。

建议：

1. 在 RAG system prompt 中增加硬约束：只允许基于检索到的文档片段作答。
2. 当来源不足时，明确输出“不确定”或“文档中未找到足够依据”。
3. 要求模型优先引用来源片段中的关键词、术语和结论。
4. 对于高风险问题，优先返回保守答案，而不是补全推断。

建议优先级：高

### 4.2 检索质量

目标是让模型真正拿到更有价值的上下文，而不是只做形式上的向量检索。

建议：

1. 重新评估 chunk 大小和 overlap 是否适合当前文档类型。
2. 对不同文档格式做检索效果对比，观察过细或过粗切分带来的影响。
3. 试验不同 top-k 参数，避免召回过少漏信息，或召回过多稀释重点。
4. 保留关键词检索兜底，并考虑混合排序而不是二选一切换。

建议优先级：高

### 4.3 多轮对话

目标是让追问稳定可用，同时控制成本和延迟。

建议：

1. 限制参与 prompt 的历史轮数，只保留最近有效上下文。
2. 当对话过长时，对旧消息进行摘要而不是全部拼接。
3. 对重新生成、追问、纠错等场景保留必要的用户意图信息。
4. 区分“系统提示”“检索结果”“历史对话”三类上下文，避免互相污染。

建议优先级：中高

### 4.4 文档状态

目标是让用户知道文档当前处于哪个阶段，以及什么时候可以开始问答。

建议：

1. 后端统一定义状态枚举：上传中、解析中、可用、失败。
2. 前端列表和详情页展示一致的状态标签和文案。
3. 失败状态应携带简短原因，帮助用户理解问题。
4. 只有处于“可用”状态的文档才允许进入问答范围。

建议优先级：中高

### 4.5 Provider 可用性

目标是让用户配置 provider 时更少踩坑。

建议：

1. 在设置页明确区分聊天模型和 embedding 模型。
2. 对不支持 embedding 的 provider 直接显示说明，而不是让用户猜。
3. 对 `enable_embedding` 开关配上清楚的提示文案。
4. 若系统将退回关键词检索，应在 UI 中明确提示当前行为。

建议优先级：中高

### 4.6 可观测性

目标是让后端问题可定位、可解释，而不是只能靠猜。

建议：

1. 增加 provider 调用日志，记录当前使用的是哪个 provider 和模型。
2. 增加检索日志，记录命中的 chunk 数量、来源文档和排序结果。
3. 当 embedding 不可用而回退关键词检索时，明确记录回退原因。
4. 对 SSE 中断、模型超时、向量生成失败等场景做统一日志输出。

建议优先级：中

### 4.7 数据安全

目标是避免系统在演示或交付时因为密钥处理方式而被质疑。

建议：

1. 优先考虑对 `api_key` 做加密存储。
2. 如果当前阶段暂不做加密，也应在文档中明确说明数据安全边界。
3. 避免在日志、接口响应和前端页面中泄露完整密钥。
4. 在 README 或技术文档中单独加一段说明当前安全策略和后续改进方向。

建议优先级：中

### 4.8 部署体验

目标是让第三方可以快速启动项目并完成基本验收。

建议：

1. 在 README 中增加“一条命令启动”的最短路径说明。
2. 写清楚启动后如何验证前端、后端和代理链路都正常。
3. 给出最短的验收流程，例如上传文档、配置 provider、创建问答会话、查看来源。
4. 对 Docker 和本地开发两条启动路径分别给出步骤。

建议优先级：中

## 5. 推荐实施顺序

如果后续时间有限，建议按以下顺序推进：

1. 问答准确性
2. 检索质量
3. 文档状态
4. Provider 可用性
5. 多轮对话
6. 可观测性
7. 部署体验
8. 数据安全

## 6. 一句话总结

当前项目已经具备完整的主流程能力，下一阶段最重要的不是继续堆功能，而是围绕问答质量、系统稳定性、用户可理解性和交付体验做针对性优化。
