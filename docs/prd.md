# 智能文档问答助手 PRD

## 1. 项目背景与目标

### 1.1 项目背景

团队需要构建一个面向文档知识检索场景的智能问答系统，支持用户上传多种格式文档，并通过自然语言提问快速获取答案。该系统用于课程项目、技术作业或内部演示场景，重点在于实现一个可运行、可演示、可验收的 MVP，而不是追求复杂的企业级平台能力。

### 1.2 项目目标

本项目需要交付一个前后端分离的智能文档问答系统，满足以下目标：

1. 支持用户上传并管理 `.txt`、`.md`、`.markdown`、`.pdf`、`.docx` 文档。
2. 支持基于已上传文档内容进行问答，并返回准确、连贯的回答。
3. 支持多轮对话，能够结合当前会话上下文继续回答追问。
4. 支持配置和切换不同 AI 服务提供商，包括 OpenAI、Claude 和兼容 OpenAI API 的本地模型。
5. 提供统一、完整、可演示的 Web UI，覆盖文档管理、Agent 问答和系统配置三个核心页面。
6. 提供可运行镜像、本地联调方式、基础测试及完整项目文档，满足交付要求。

### 1.3 成功标准

项目完成后，应满足以下结果：

1. 用户可在页面中上传文档、查看文档、删除文档。
2. 用户可针对文档发起问答，并在多轮对话中持续追问。
3. 管理员可配置不同模型服务商，并切换默认问答模型。
4. 系统可通过 Docker Compose 启动并正常运行。
5. 项目具备可验收的代码、测试、接口和部署文档。

## 2. 产品范围与版本边界

### 2.1 MVP 范围

本期必须完成以下范围：

1. 文档上传、解析、存储、列表查询、删除。
2. 基于文档内容的问答能力。
3. 多轮对话能力。
4. AI 服务商配置与默认模型切换。
5. 文档管理页、Agent 问答页、系统配置页。
6. 问答响应支持 SSE 流式输出，前端逐字渲染。
7. Agent 问答页支持会话历史侧栏，可浏览、切换和删除历史会话。
8. 问答消息支持 Markdown 格式渲染（标题、列表、代码块等）。
9. 前后端分离架构。
10. 可运行镜像与本地联调方案。
11. 基础单元测试与后端接口测试。
12. 架构设计文档、使用说明、接口文档、部署文档。

### 2.2 明确不在 MVP 范围内

以下能力不属于本期必做项：

1. 用户登录、权限控制、多租户。
2. 审批流、复杂工作流编排。
3. 移动端 App。
4. 文档协作编辑。
5. 企业级监控告警平台集成。

### 2.3 扩展能力与加分项

以下内容作为扩展能力或加分项，可在 MVP 完成后追加：

1. 支持答案引用来源片段和高亮。
2. 支持更完整的向量召回策略优化。
3. 支持本地离线部署模型的增强能力。
4. 支持深色模式与主题切换。

## 3. 用户角色与典型场景

### 3.1 用户角色

1. 普通用户：上传文档、查看文档、发起问答、进行多轮会话。
2. 系统管理员：配置模型服务商、维护默认模型、测试服务连通性。

### 3.2 典型场景

#### 场景一：文档上传与管理

用户上传项目说明文档或知识库文档，系统完成解析后在文档列表中展示状态，用户后续可查看文档信息或删除文档。

#### 场景二：基于文档问答

用户上传文档后提问“这个系统支持哪些文件格式”，系统基于文档内容作答，而不是脱离文档自由发挥。

#### 场景三：连续追问

用户先提问“系统支持哪些模型供应商”，再继续追问“默认应该选哪个”，系统应结合当前会话上下文理解追问对象。

#### 场景四：系统配置

管理员在系统配置页填写 OpenAI、Claude 或本地兼容模型的接口地址、模型名和密钥，并切换默认 Provider 供后续新会话使用。

## 4. 核心业务流程

### 4.1 文档上传与入库流程

1. 用户在文档管理页选择文件上传。
2. 前端校验文件类型和基础信息。
3. 后端接收文件并保存原始文件与元数据。
4. 系统对文档进行解析。
5. 文档状态按规则流转为“上传中 / 解析中 / 可用 / 失败”。
6. 解析成功的文档进入可问答范围。
7. 解析失败时，页面需展示失败状态和错误提示。

### 4.2 问答流程

1. 用户进入 Agent 问答页，可从会话历史侧栏选择已有会话或新建会话。
2. 用户选择问答范围，支持”全部文档”或”单个文档”。
3. 用户输入问题并发送。
4. 后端结合当前会话历史和文档内容生成回答，通过 SSE 流式输出。
5. 前端逐字渲染回答内容，并以 Markdown 格式展示。
6. 若文档中无法找到依据，系统必须明确返回”未在文档中找到依据”或语义等价表述。
7. 回答完成后在消息流中展示本轮提问与回答。

### 4.3 模型配置与切换流程

1. 管理员进入系统配置页。
2. 填写或修改 Provider 配置。
3. 执行连接测试。
4. Provider 保存后默认处于未验证状态，需先执行连接测试；测试成功后方可设置默认 Provider。
5. 新建会话使用最新默认 Provider；已在进行中的会话继续使用会话绑定的 Provider，MVP 默认不强制切换历史会话。

## 5. 功能需求

### 5.1 文档管理

#### 5.1.1 文档上传

系统应支持上传以下主交付格式：

1. `.txt`
2. `.md`
3. `.markdown`
4. `.pdf`
5. `.docx`

上传相关要求如下：

1. 页面需提供单文件上传入口。
2. 后端需校验文件格式，不允许上传非指定格式。
3. 单文件大小上限为 50 MB，超出时前后端均需给出明确提示。
4. 系统需记录文档名称、文件类型、文件大小、上传时间、解析状态。
5. 上传成功后，文档需立即出现在文档列表中。
6. 文件不合法时，系统需给出明确失败提示。
7. 允许上传同名文件，每次上传创建独立文档记录。

#### 5.1.2 文档解析与状态流转

系统应在上传后完成文档解析，并基于解析结果维护文档状态。

状态要求如下：

1. `上传中`：文件已接收并完成基础保存，等待后台任务接手处理。
2. `解析中`：系统正在提取文档文本内容。
3. `可用`：文档解析成功，可参与问答。
4. `失败`：解析未成功，当前文档不可参与问答。

业务规则如下：

1. 只有 `可用` 状态的文档可以被问答命中。
2. `失败` 的文档需展示失败状态。
3. 解析失败不应导致其他文档不可用。
4. MVP 阶段至少支持 TXT、Markdown、PDF、Word 的文本提取和基础元数据存储。
5. 系统需检测停滞的解析任务：若任务处于 `运行中` 状态超过 5 分钟，自动重置为待处理状态以触发重试。
6. 文档内容为空（0 字节或提取文本为空）时，应标记为 `失败` 并提示"文档内容为空"。

#### 5.1.3 文档查询

系统应支持用户查询和浏览已上传文档。

要求如下：

1. 文档列表需展示名称、类型、大小、上传时间、状态。
2. 支持按文档名称搜索。
3. 支持分页展示。
4. 支持查看文档基础详情。

#### 5.1.4 文档删除

系统应支持删除文档及其关联问答数据依赖。

要求如下：

1. 页面需提供删除操作入口。
2. 删除前需弹出确认提示。
3. 删除后，该文档不应继续出现在列表中。
4. 删除后，该文档不应再参与问答命中。

### 5.2 Agent 问答

#### 5.2.1 单轮问答

系统应支持用户发起自然语言提问，并针对文档内容生成回答。

要求如下：

1. 用户可在 Agent 问答页输入问题。
2. 用户可选择问答范围为”全部文档”或”单个文档”。
3. 系统回答应尽量基于文档事实，而不是泛化生成。
4. 回答应保证语言连贯、语义清晰。
5. 无法从文档中获得答案时，必须明确告知用户未找到依据，不能编造答案。
6. 回答内容采用 SSE（Server-Sent Events）流式输出，前端逐字渲染，提升响应体验。
7. 回答内容支持 Markdown 格式渲染，包括标题、列表、代码块、加粗等。

#### 5.2.2 多轮对话

系统应支持多轮会话，保持当前对话上下文连续性。

要求如下：

1. 系统需记录当前会话的问答历史。
2. 用户可基于上轮问题继续追问。
3. 页面需支持新建会话。
4. 页面需提供会话历史侧栏，展示历史会话列表，支持切换、删除历史会话。
5. 会话标题根据首次提问内容自动生成。
6. MVP 默认保留当前会话历史，不要求实现复杂的跨会话知识合并。

#### 5.2.3 回答质量原则

系统在回答时应遵循以下原则：

1. 优先依据文档内容回答。
2. 减少模型幻觉。
3. 对证据不足的问题给出明确提示。
4. 对连续追问保持主题一致性。

### 5.3 系统配置

#### 5.3.1 Provider 管理

系统配置页至少应支持以下三类模型服务：

1. OpenAI
2. Claude
3. 兼容 OpenAI API 的本地模型

每个 Provider 至少支持以下配置项：

1. Provider 类型
2. 接口地址
3. 模型名称
4. API Key
5. 温度参数
6. 最大 Token
7. 超时时间

#### 5.3.2 Embedding 服务说明

系统使用 Provider 级 Embedding 配置生成文档和查询向量，优先跟随当前会话绑定的 Provider。

规则如下：

1. 每个 Provider 可独立配置 `enable_embedding` 和 `embedding_model`。
2. 问答检索优先使用当前会话绑定的 Provider 生成查询向量。
3. 若该 Provider 不支持 Embedding 或调用失败，系统自动回退为关键词检索。
4. 文档切片向量按 `provider_id + embedding_model` 隔离存储，避免不同向量空间混用。

#### 5.3.3 默认模型切换

系统应支持切换默认问答模型。

规则如下：

1. 管理员保存配置后，Provider 默认为未验证状态，需先通过连通性测试。
2. 新建问答会话默认使用当前默认 Provider。
3. 若 Provider 测试失败或尚未验证，系统需阻止设置为默认值并给出提示。

#### 5.3.4 连通性测试

系统应支持对当前 Provider 配置执行连接测试。

要求如下：

1. 页面需提供“测试连接”操作。
2. 测试成功时给出成功反馈。
3. 测试失败时给出错误反馈。
4. 测试失败不应静默保存为可用配置。

## 6. 页面与交互需求

### 6.0 全局设计规范

#### 组件库

前端统一使用 Ant Design (antd) 组件库，利用其内置的中文 locale、数据密集型组件和设计体系。

#### 导航结构

采用水平顶部导航栏：
- 左侧：应用名称/Logo
- 右侧：三个导航链接（文档管理、Agent 问答、系统配置）
- 当前页面高亮显示

#### 默认路由

根路由 `/` 重定向到 `/documents`（文档管理页）。首次使用的用户需要先上传文档才能开始问答。

#### 统一错误展示模式

系统采用三级错误展示：
1. **表单校验错误**：字段下方行内红色提示文字（Ant Design Form 内置）。
2. **异步操作失败**（上传/删除/保存/API 调用）：右上角 Toast 通知（Ant Design notification），5 秒自动消失。
3. **SSE/流式输出失败**：消息流中内联错误提示，附带 [重试] 按钮。

#### 响应式设计

系统需支持全响应式布局：

| 视口 | 文档管理页 | Agent 问答页 | 系统配置页 |
| --- | --- | --- | --- |
| 桌面（≥1280px） | 完整表格，所有列 | 侧栏 + 消息区域 | 列表 + 行内编辑 |
| 平板（768-1279px） | 表格隐藏大小列 | 侧栏收起为抽屉（☰） | 列表/表单上下堆叠 |
| 手机（<768px） | 堆叠卡片（名称+状态+操作） | 全屏消息，会话列表为抽屉 | 堆叠表单，每次显示一个 Provider |

#### 无障碍要求

1. 所有交互元素支持键盘导航（Tab、Enter、Escape）。
2. 页面区域使用 ARIA 语义标签（`<nav>`、`<main>`、`<aside>`）。
3. 文字对比度符合 WCAG AA 标准（正文 4.5:1，UI 组件 3:1）。
4. 移动端触摸目标最小 44×44px。
5. 所有可聚焦元素显示 focus-visible 轮廓。
6. 状态标签不仅用颜色区分，还需包含文字标签（如"可用"、"失败"）。

### 6.1 文档管理页

#### 页面布局

```
┌──────────────────────────────────────────────────┐
│  DocQA    文档管理 | Agent 问答 | 系统配置        │ ← 顶部导航
├──────────────────────────────────────────────────┤
│  ┌─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─┐  │
│  │  📤 点击或拖拽文件到此处上传                 │  │ ← 拖拽上传区
│  │  支持 .txt .md .pdf .docx，单文件 ≤50MB    │  │   (Ant Design Upload.Dragger)
│  └─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─┘  │
│                                                  │
│  搜索: [________________]                        │ ← 搜索框
├──────────────────────────────────────────────────┤
│  名称      │ 类型 │ 大小   │ 上传时间 │ 状态  │   │ ← 表格布局
│  ──────────┼──────┼────────┼─────────┼───────┼───│
│  report.pdf│ PDF  │ 2.1 MB │ 03/23   │ 🟢可用 │ 🗑│
│  notes.txt │ TXT  │ 5.3 KB │ 03/23   │ 🟡解析中│ 🗑│
│  spec.md   │ MD   │ 12 KB  │ 03/22   │ 🔴失败 │ 🗑│
├──────────────────────────────────────────────────┤
│  ← 1  2  3  →                                   │ ← 分页器
└──────────────────────────────────────────────────┘
```

#### 核心组件

1. **拖拽上传区域**（Ant Design Upload.Dragger）：支持点击选择文件或拖拽上传，显示支持的格式和大小限制。
2. **文档表格**（Ant Design Table）：列包括名称、类型、大小（人类可读格式如 KB/MB）、上传时间、状态标签、操作。
3. **搜索框**：按文档名称搜索过滤。
4. **分页器**（Ant Design Pagination）。
5. **删除确认弹窗**（Ant Design Modal）：显示文档名称，"确认删除？此操作不可恢复"，取消 + 删除（红色）按钮，删除按钮在 API 调用期间显示加载状态。
6. **状态标签**（Ant Design Tag）：
   - 灰色 = 上传中
   - 琥珀色/黄色（带脉冲动画）= 解析中
   - 绿色 = 可用
   - 红色 = 失败（悬停显示 error_message 工具提示）

#### 前端轮询策略

当任意文档处于 `上传中` 或 `解析中` 状态时，前端每 3 秒轮询 `GET /api/documents` 刷新状态。所有文档进入终态（可用/失败）后停止轮询。使用 TanStack Query 的 `refetchInterval` 实现。

#### 主要操作

1. 拖拽或点击上传文档（上传期间按钮显示加载状态，成功后新行先以"上传中"状态出现，再转为"解析中"）
2. 搜索文档
3. 查看文档基础信息
4. 删除文档（需确认）

#### 文件大小展示

统一使用人类可读格式，保留一位小数：如 "2.1 MB"、"15.3 KB"。

#### 异常与空状态

1. **上传格式不合法**：行内提示（Toast）显示"不支持的文件格式，请上传 .txt/.md/.pdf/.docx 文件"。
2. **上传失败**：Toast 通知显示失败原因。
3. **文件超过 50MB**：上传前客户端校验，Toast 提示"文件大小超过 50MB 限制"。
4. **文档解析失败**：列表中显示红色失败标签，悬停查看错误详情。
5. **列表为空**（零文档）：居中显示图标 + "暂无文档，上传文档即可开始智能问答" + [📤 上传文档] 主操作按钮。
6. **搜索无结果**：居中显示"未找到匹配的文档，请尝试其他关键词"。

#### 页面布局

#### 核心组件

1. **会话历史侧栏**（宽度 256px，平板端收起为抽屉）：含会话列表（按创建时间倒序）、[+ 新建会话] 按钮、每条会话支持删除操作。会话标题单行截断并显示省略号，悬停显示完整标题工具提示。
2. **消息流区域**：采用文档式布局（非气泡式），全宽消息区域，每条消息左侧显示角色图标（👤 用户 / 🤖 助手）。助手消息支持 Markdown 渲染（react-markdown + rehype-sanitize），用户消息显示为纯文本。流式输出时在文本末尾显示闪烁光标（▌）作为加载指示器。
3. **提问输入框**：支持多行输入（自动扩展 textarea），Enter 发送，Shift+Enter 换行。
4. **发送按钮**：流式输出期间变为禁用状态。
5. **会话头部**：显示当前会话范围（只读标签，如"全部文档"或具体文档名称）。

#### 新建会话流程

点击 [+ 新建会话] 后弹出创建对话框：
- 范围选择：单选按钮 (●) 全部文档 / (○) 单个文档
- 单文档模式下显示文档下拉选择器（仅显示"可用"状态文档）
- [创建会话] 按钮
- 创建后，范围以只读标签形式显示在会话头部

#### 主要操作

1. 发起提问（输入问题并发送）
2. 继续追问
3. 新建会话（通过创建对话框选择范围）
4. 浏览和切换历史会话
5. 删除历史会话（需确认）

#### 异常与空状态

1. **流式输出中**：消息末尾显示闪烁光标（▌），发送按钮禁用。
2. **模型调用失败**：消息流中内联显示错误提示 + [重试] 按钮。
3. **无可用文档**：消息区域居中显示"暂无可用文档，请先上传文档" + [前往上传] 链接跳转到 /documents。
4. **未检索到依据**：返回"⚠ 未在文档中找到依据。建议：• 尝试用不同方式描述您的问题 • 上传更多相关文档 [去上传]"。
5. **SSE 连接中断**：消息流中内联显示"⚠ 连接中断" + [重试] 按钮。
6. **零会话**（首次使用）：侧栏显示 [+ 新建会话] 按钮，消息区域居中显示"开始一段新对话，向您的文档提问" + [新建会话] 按钮。
7. **新会话零消息**：居中显示"向已上传的文档提问" + 当前范围标签 + 输入框处于可用状态。

## 7. 技术约束与系统架构

### 7.1 强约束技术选型

本项目默认技术栈固定如下：

1. 前端：React + TypeScript
2. 后端：Python + FastAPI
3. 数据库：SQLite（开发与演示默认）
4. 向量能力：ChromaDB 本地向量库
5. Embedding：跟随 Provider 配置，优先使用会话绑定 Provider 的 `embedding_model`，失败时回退关键词检索
6. 流式输出：Server-Sent Events (SSE)
7. 容器化：Docker + Docker Compose
8. 测试：前端单元测试、后端单元测试、后端接口测试

本期 PRD 不再保留 React/Vue、Python/Go 等开放式二选一表述。

### 7.2 系统架构要求

系统应采用前后端分离架构，并至少包含以下模块：

1. 前端应用模块：负责页面展示、状态管理和用户交互。
2. 文档管理模块：负责上传、解析、查询、删除。
3. 问答服务模块：负责文档检索、上下文组织和模型调用。
4. 系统配置模块：负责 Provider 配置管理、默认模型切换、连通性测试。
5. 数据存储模块：负责文档元数据、会话数据、配置数据的存储。

### 7.3 数据与部署要求

1. 文档原文和元数据需有清晰存储方案。
2. 文档向量或检索相关数据应存储在 ChromaDB 本地向量库中，并与 SQLite 元数据存储配合使用。
3. 项目需提供可构建镜像。
4. 项目需提供 Docker Compose 本地联调方案。

## 9. 验收标准

### 9.1 功能验收标准

1. 上传 `.txt`、`.md`、`.markdown`、`.pdf`、`.docx` 后，文档可在列表中展示，并在解析成功后进入 `可用` 状态。

3. 删除文档后，该文档不再出现在列表中，也不会继续参与问答命中。

## 11. 风险与后续迭代

### 11.1 主要风险

1. 模型回答偏离文档内容，导致问答可信度下降。
2. 多 Provider 接入协议差异较大，增加适配复杂度。
3. PDF、Word 的解析质量可能受文档结构、扫描内容或格式兼容性影响。
4. 多轮对话上下文过长可能带来响应时延和成本上升。

### 11.2 应对策略

1. 问答服务优先以文档检索结果作为回答依据。
2. 通过统一 Provider 抽象减少不同模型 API 差异影响。
3. 优先保证 TXT、Markdown、PDF、Word 的稳定解析，再继续优化复杂版式和兼容性。
4. 通过会话上下文裁剪控制模型调用长度。

### 11.3 后续迭代方向

1. 增加答案引用来源片段展示。
2. 增加更细粒度的检索和召回优化。
3. 增加扫描版 PDF、复杂表格和图片内容的增强解析能力。
4. 增加用户体系和权限管理能力。
